# 慧阅——图书馆 AI 低碳智控系统

## 摘  要

随着我国"碳达峰、碳中和"战略的深入推进，建筑领域的绿色低碳转型已从宏观政策转化为具体场景中的技术需求。高校图书馆因使用频次高、空调与照明能耗集中，是校园碳能管理中节能潜力最大的建筑类型之一。然而，当前图书馆环境管控普遍面临空调按固定时间表运行导致无人时段能耗浪费、照明缺乏按需调节机制、安全管控缺少自动化手段三大痛点。现有楼宇能耗管理系统多停留在数据监测阶段，尚未形成从感知到自主控制的闭环；大语言模型在实时控制中面临延迟不可控、输出不确定和网络依赖三重挑战。如何在保证控制确定性的前提下将 AI 决策嵌入楼宇环境管控闭环，是图书馆低碳化改造的关键问题。

针对上述问题，本文设计并实现了慧阅——一套基于双引擎 AI 决策架构的图书馆低碳智控系统，核心创新体现在两个层面。在控制策略上，摒弃单一 AI 直接控制硬件的范式，采用"本地规则引擎为主、LLM 顾问为辅"的协作架构：引擎一为多传感器加权融合的态势感知状态机，以 CO₂、噪声及环境变化率五维数据实时计算空间占用态势，实现零 API 成本的毫秒级响应；引擎二为大语言模型顾问引擎，仅在有人状态下按需激活，综合传感器、天气与偏好数据进行推理，所有建议经本地校验后由规则引擎执行。双引擎架构将 AI 的推理深度与工业控制的确定性解耦，使系统既具备多因素自适应优化能力，又不牺牲实时控制的可靠性与故障韧性。

系统基于霍尼韦尔 Niagara Framework 4 + JACE8000 + IO-22U 工业物联网技术栈实现，传感器与执行器通过 4-20mA 工业标准信号硬接线接入 I/O 模块，经 Modbus RTU 与 oBIX 两层开放协议贯通至 PC 端 AI 引擎，前端提供 B/S 架构的 Dashboard 可视化平台。CO₂ 加权融合采用非对称滞回设计，利用气体扩散的物理惰性天然抑制传感器噪声引发的状态翻转；安全告警规则在边缘控制器与 PC 端双重独立执行、互为冗余。系统以空间占用状态为调度核心，无人时段强制休眠、有人时段按需供能，将"人"的在场信息作为能耗决策的根本判据，为高校图书馆提供了一套兼具 AI 自适应能力与工业级可靠性的低碳管控解决方案。

**关键词**：Niagara Framework；双引擎 AI 架构；态势感知状态机；大语言模型；图书馆低碳管控

## 第一章  设计需求分析

### 1.1 背景及意义

随着我国"碳达峰、碳中和"战略的深入推进，建筑领域作为能耗和碳排放的三大重点领域之一，面临着迫切的绿色低碳转型需求。高校作为人才培养和科技创新的重要基地，其校园建筑的低碳化改造既是响应国家战略的切实行动，也是培养学生环保意识、推广绿色理念的有效途径。图书馆作为高校的核心公共建筑，使用频次高、服务时间长、设备能耗集中，是校园碳能管理的优先切入点。

然而，高校图书馆的能耗管理长期面临三大核心痛点。其一，**空调浪费严重**。阅览区人员密度随时间剧烈波动——上课时段人员稀疏、考试季座无虚席、寒暑假几乎空置——但空调系统多按固定时间表全天恒定运行，无人时段能耗浪费突出，空调能耗通常占图书馆总能耗的 40% 以上。其二，**照明粗放管理**。靠窗区域自然光充足但灯光常开，闭架书库无人也维持全亮状态，缺乏按需照明的自适应调节机制，照明能耗通常占图书馆总能耗的 20% 至 30%。其三，**静音与安全管控缺失**。阅览区缺乏客观的噪声监测手段，安静管理完全依赖人工巡视；同时缺乏对烟雾积聚、CO₂ 浓度过高等安全隐患的自动检测与告警能力。

针对上述痛点，本项目基于霍尼韦尔 Niagara Framework 4 + JACE8000 边缘控制器 + VYKON IO-22U 扩展 I/O 模块技术栈，构建面向高校图书馆的 AI 自主优化低碳管控系统。项目全面贴合霍尼韦尔赛道"AI 自主优化低碳数智化城市更新"核心命题：在技术生态上深度集成 Niagara 平台与 oBIX 通信协议，在硬件生态上实现 JACE8000 + IO-22U 全链路闭环，在智能控制上以双引擎 AI 架构实现按需供能与无人休眠的节能策略，在场景选择上锚定需求明确、痛点清晰、易于验证的高校图书馆细分场景。

### 1.2 研究近况

随着物联网、人工智能与建筑自动化技术的深度融合，智慧楼宇的低碳管控已从单一的能耗监测向多源感知与自主决策的方向演进。校园碳能管理领域在数据采集层面趋于成熟，但在从"监测"到"自主控制"的闭环方面仍存在不足；大语言模型的快速发展为物联网智能决策开辟了新路径，但将其安全地嵌入实时控制系统中仍是一个开放课题；Niagara Framework 作为成熟的楼宇集成平台，在设备互联互通方面具有显著优势，而 CO₂ 等多传感器融合的态势感知技术则为空间占用检测提供了物理上可靠的判断依据。以下分别就上述四个领域的研究现状进行综述。

#### 1.2.1 校园碳能管理研究

据中国建筑节能协会《中国建筑能耗研究报告（2023）》，我国公共建筑单位面积能耗约为居住建筑的 2-3 倍，其中高校图书馆因使用频次高、服务时间长、设备能耗高度集中，是校园碳能管理中节能潜力最大、改造需求最迫切的建筑类型之一。在技术层面，近年来校园能耗管理研究主要沿两条路径展开：一是基于物联网传感器网络和建筑信息模型（BIM）的能耗监测平台建设，实现水、电、暖通空调的分项计量与可视化展示；二是基于历史能耗数据的统计分析和机器学习建模，通过聚类、回归等方法识别用能异常模式并生成节能建议。在此基础上，部分前沿研究引入了模型预测控制（MPC）和深度强化学习等先进方法，尝试在仿真环境中对空调和照明系统进行动态优化调度。然而，从整体来看，现有研究在数据采集和可视化层面已趋于成熟，但在从"监测"到"自主控制"的闭环方面仍存在明显短板：多数系统停留在数据展示和事后分析阶段，缺乏接入多源传感器数据后实时生成控制策略并自动下发执行的能力，且对通信中断、设备故障等异常工况下的系统韧性和控制连续性关注不足。

#### 1.2.2 大语言模型在物联网中的应用

以 GPT-4、DeepSeek 为代表的大语言模型（Large Language Model, LLM）在 2023-2024 年间取得了突破性进展，其在自然语言理解、多模态推理和代码生成等方面的能力迅速从实验室走向产业应用。在物联网领域，学术界与工业界开始探索将 LLM 作为智能决策层的核心组件，利用其强大的情境推理能力综合异构数据源——包括传感器读数、用户指令、天气预报和历史运行记录——直接生成结构化的设备控制策略。代表性工作包括利用 GPT-4 进行家庭能源管理的自然语言交互控制、结合检索增强生成（RAG）技术提升工业设备故障诊断的准确率、以及通过 Function Calling 机制将 LLM 输出约束为可被物联网平台直接调用的 API 指令序列等。然而，LLM 在物联网实时控制场景中的落地面临三个关键挑战：第一，云端 API 调用的网络延迟（通常在数百毫秒到数秒量级）和按 Token 计费的成本模型难以满足秒级甚至毫秒级实时控制的需求；第二，LLM 的生成结果具有概率性，可能输出不符合物理约束或安全规范的错误指令，直接控制硬件设备存在安全风险；第三，LLM 依赖网络连接，在网络中断时完全不可用，这对于需要"断网不断控"的工业级物联网系统是不可接受的。这些挑战驱动了本系统"LLM 顾问 + 本地规则引擎"的双引擎架构设计。

#### 1.2.3 Niagara Framework 在智慧楼宇中的应用

Niagara Framework 是霍尼韦尔旗下 Tridium 公司开发的开放式物联网软件平台，自 1996 年推出以来已迭代至第四代（Niagara 4），在全球超过 80 个国家和地区拥有数百万个部署实例。Niagara 的核心价值在于其多协议兼容能力——通过内置的驱动框架，Niagara 可以无缝集成 BACnet、Modbus、LonWorks、OPC UA、MQTT 等数十种主流工业通信协议，解决楼宇自动化领域长期存在的"异构系统互联互通"难题。在智慧楼宇场景中，基于 Niagara 的解决方案已在悉尼歌剧院低碳智联改造、保利地产集团总部智慧能源管理、厦门智慧管廊综合监控等大型项目中得到验证。其边缘控制器 JACE8000 搭载 QNX 实时操作系统和 ARM Cortex-A8 处理器，能够在不依赖云端的情况下于边缘端独立运行 Wire Sheet 图形化控制逻辑，确保关键控制功能的确定性执行。近年来，随着 AI 与物联网的深度融合，如何将 Niagara 强大的设备集成能力与大语言模型的智能推理能力相结合，构建既有工业级可靠性又具备 AI 自适应优化能力的新一代楼宇控制系统，已成为学术界和产业界共同关注的课题。

#### 1.2.4 态势感知技术在空间占用检测中的应用

空间占用检测（Occupancy Detection）是实现建筑按需环境控制的前提——只有在准确判断"空间里是否有人、有多少人"的基础上，系统才能做出合理的空调、照明和新风控制决策。在传感器选择方面，CO₂ 浓度因人体新陈代谢持续产生且气体扩散具有物理惰性（浓度不会随人员短暂离开而瞬间下降），被 ASHRAE Standard 62.1-2022 等国际标准推荐为室内人员密度的间接估计指标。相比于被动红外（PIR）传感器仅能检测运动目标、无法感知静坐人员的固有缺陷——这在图书馆阅览区等以安静阅读为主的场景中是致命的——CO₂ 传感器可以覆盖人员静坐、学习等低活动量场景。在数据融合方法方面，基于贝叶斯推理、卡尔曼滤波和加权线性融合的多传感器综合策略已被广泛研究，不同传感器按其物理可靠性赋予差异化权重。CO₂ 浓度凭借其与人员存在之间最为直接的物理因果关系，在多传感器融合中通常被赋予最高权重。本系统在此基础上将温度变化率、湿度变化率和光照变化率等派生特征纳入融合框架，通过感知人员活动引起的环境微变进一步提升检测灵敏度。

### 1.3 图书馆场景碳能管控的设计需求

综合上文对高校图书馆能耗痛点的分析、四个交叉领域研究现状的梳理以及本系统在霍尼韦尔赛道中的定位，一个面向图书馆场景的 AI 低碳智控系统需要在以下六个维度上满足设计要求。这六项需求由底层数据采集向上层用户体验逐级递进，覆盖了从环境感知、状态认知、智能决策、安全保障到人机交互和架构可持续性的完整物联网系统链路。以下逐一阐述。

1. **实时数据采集**。系统需以不低于 2 秒的周期采集图书馆内的核心环境参数，包括温度、湿度、光照度、CO₂ 浓度、噪声水平、烟雾浓度和 PM2.5 浓度，并同步采集风扇回路的实时电流以计算功率。所有传感器应采用工业级 4-20mA 电流信号输出，通过 IO-22U 的 12 位模数转换器完成模拟量到数字量的转换，再经 Modbus RTU 总线上传至 JACE8000。数据采集通道需具备独立的在线状态监测能力，单个传感器故障不应影响其余通道的正常采集。

2. **智能分析管理**。系统不能仅仅展示传感器数值，而需要从多维环境数据中提取更高层次的空间状态信息。具体而言，系统需基于 CO₂ 浓度、噪声水平、温湿度变化率和光照变化率等多源数据，采用加权融合算法实时计算表征空间人员占用程度的综合态势值，并据此驱动有限状态机（FSM）在空闲、过渡、占用三种状态之间迁移。FSM 的状态判定需具备非对称滞回特性——进入占用态的阈值应高于退出占用态的阈值——以利用 CO₂ 等物理量的消散惰性抑制传感器噪声引发的频繁状态翻转，确保状态输出的稳定性。

3. **智能控制优化**。在准确判断空间占用状态的基础上，系统需根据状态差异执行不同的控制策略：空闲状态下强制关闭照明和风扇以最大限度节能；占用状态下根据当前光照度、温度和 CO₂ 浓度，结合用户选择的偏好模式（节能优先、均衡模式或舒适优先），自动计算最优的照明亮度和风扇启停方案。此外，系统需引入大语言模型作为智能顾问层——在占用状态下按需激活 LLM，综合室内传感器数据、室外天气信息和用户偏好进行深度推理，输出可解释的环境优化建议。但 LLM 的输出仅作为参考输入提供给本地规则引擎，不直接控制任何硬件设备，以确保 AI 参与决策但不超过安全边界。

4. **系统安全韧性**。图书馆作为人员密集的公共建筑，其环境控制系统必须具备应对传感器故障、通信中断和 AI 服务不可用等异常工况的能力。烟雾超标和噪声持续超标两项安全告警功能必须独立于 AI 决策链路和 FSM 舒适性控制逻辑运行，在 JACE8000 本地 Wire Sheet 图形化逻辑和 PC 端 Python 规则引擎中双重实现——即使 PC 端完全宕机，JACE8000 仍能独立完成安全告警的触发与解除。AI 决策链路（LLM 顾问）故障时，系统需自动降级为纯 FSM 本地规则模式，照明和风扇的基础控制功能不中断。告警规则需引入完整的回滞区间（防止边界抖动导致的反复触发）和持续时间判据（防止瞬时干扰导致的误报），确保告警输出的可靠性。

5. **用户界面交互**。系统需提供基于 B/S 架构的 Web 可视化 Dashboard，运行于 PC 端本地服务器，评委与观众通过浏览器即可访问。Dashboard 需至少包含以下功能视图：主控看板（传感器数据实时卡片、FSM 状态指示灯、设备状态面板、LLM 决策文字和告警信息一览）、AI 对话助手（支持快捷提问和流式响应）、历史趋势分析（多传感器折线图叠加、时间范围切换和数据导出）、能耗对比分析（AI 模式与常开模式的能耗差异量化展示）。

6. **系统扩展兼容性**。软件架构需遵循模块化设计原则，核心控制逻辑——FSM、安全规则和 LLM 顾问——与具体的传感器和执行器点位配置解耦。当需要增加新的传感器类型或替换执行器设备时，仅需修改配置层中的点位映射表和量程参数，无需改动控制逻辑代码。

## 第二章  特色与创新

"慧阅"系统的构建，融合了态势感知与大语言模型推理技术，旨在打造一个兼具智能决策能力与工业级可靠性的图书馆低碳管控体系。本系统基于霍尼韦尔 Niagara Framework 4 + JACE8000 + IO-22U 技术栈进行开发，不仅实现了全链路工业级硬接线传感器的实时数据采集，更通过"本地规则引擎为主体、LLM 为顾问"的双引擎架构划定了 AI 的能力边界，确保了控制指令的确定性与安全性。在感知层面，系统以 CO₂ 为核心信号源，融合噪声、温湿度变化率和光照变化率等多维数据，利用气体消散的物理惰性实现了抗干扰的空间占用状态判定。在可靠性层面，系统构建了从 LLM 顾问到 FSM 本地规则再到 JACE8000 Wire Sheet 的三层降级保底机制，任一环节失效时上层功能自动降级而基础控制不中断。此外，系统全链路采用竞赛官方技术栈，从传感器硬接线、Modbus RTU 现场总线到 oBIX over HTTPS 数据通信形成了完整的技术闭环。核心智能体现在 AI 智能决策双引擎，它通过 FSM 状态驱动的按需激活策略将 LLM 算力集中于真正需要深度推理的时段，空闲状态下无 API 开销。这些核心特色共同赋予了"慧阅"系统在智能楼宇管控领域的独特竞争优势，整体架构如图 2-1 所示。

### 2.1 双引擎 AI 认知决策架构

将大语言模型引入物联网实时控制系统的核心难题在于LLM 的输出具有概率性，同一条指令可能在不同时刻返回不同的结果，且存在产生不合理输出的可能，而硬件控制要求每次决策都必须是确定、可预期和安全的。直接将 LLM 置于控制链路的关键路径上，等于在系统中引入了一个不可预测的故障点。为解决这一矛盾，本系统设计了一种"双引擎协作"架构，将决策过程拆分为两个职责清晰、运行独立的环节：由态势感知自适应状态机（FSM）承担确定性的空间占用判断，由 LLM 承担非确定性的环境优化建议生成，二者通过本地规则引擎实现安全耦合——FSM 和规则引擎可脱离 LLM 独立运行，LLM 的输出仅作为优化参考接入规则引擎，任何时候都不会绕过规则引擎直接作用于执行器。

引擎一为态势感知自适应状态机（FSM），部署于 PC 端 Python 主控制循环中，以 2 秒为周期持续运行。FSM 的输入是经多传感器加权融合后归一化至 [0, 1] 区间的综合态势值 S_t，输出为四个离散状态之一：VACANT（空闲，S_t 持续低于 0.30）、ARRIVING（到达过渡，S_t 由低位攀升至 0.40）、OCCUPIED（占用，S_t 达到 0.45 以上）和 LEAVING（离开过渡，S_t 由 OCCUPIED 高位回落至 0.40 以下）。FSM 采用非对称滞回阈值设计——进入 OCCUPIED 态的阈值（0.45）显著高于最终退出回 VACANT 的阈值（0.30），该差值利用 CO₂ 浓度在人员离开后不会瞬间下降的物理惰性，构成了一道天然的防抖屏障：传感器噪声引起的瞬时波动不足以跨越 0.15 的滞回区间，从而有效避免了状态在 OCCUPIED 与 VACANT 之间的频繁翻转。ARRIVING 和 LEAVING 两个过渡态分别承担"防止无人环境瞬时扰动误触发"和"确认人员稳定离开后才退回 VACANT"的职责，进一步增强了状态判断的鲁棒性。FSM 全程零 API 调用，单次评估耗时在毫秒级，是整个系统控制响应的速度基线。

FSM 的态势评分基于五维加权融合公式计算：

```
S_t = Σ_{i=1}^{5} w_i · (v_i - L_i) / (H_i - L_i)
```

其中 S_t 为时刻 t 的综合态势值，经归一化映射至 [0, 1] 区间；w_i 为各维度权重；v_i 为当前传感器读数；[L_i, H_i] 为各维度的标准化量程。权重分配和各维度归一化量程如表 2-1 所示。

**表 2-1  FSM 评分权重与归一化量程**

| 维度 | 下限 L_i | 上限 H_i | 权重 w_i | 物理依据 |
|------|---------|---------|---------|---------|
| CO₂ 浓度 | 450 ppm，室外基线浓度 | 1200 ppm，明确占用状态 | 0.40 | 人体呼吸持续产生，气体消散具有物理惰性 |
| 噪声水平 | 30 dB，极安静环境 | 70 dB，明显活动状态 | 0.25 | 交谈、走动等活动声音与人员存在正相关 |
| 温度变化率 | 0°C | 1.5°C | 0.10 | 人员进入引起短暂温度波动 |
| 湿度变化率 | 0%RH | 8%RH | 0.10 | 人员呼吸和体表蒸发引起湿度微变 |
| 光照变化率 | 0 Lux | 350 Lux | 0.15 | 人员移动遮挡或开关门引起照度变化 |

权重分配遵循物理因果关系的强弱而非数据拟合：CO₂ 因其与人体新陈代谢之间直接且稳定的因果关系获配最高权重；噪声作为主动活动的直接表征次之；温湿度和光照的变化率作为辅助维度，感知人员活动引起的环境微变。这种"一核心、四辅助"的融合结构确保当单个辅助传感器因故障出现异常读数时，评分体系的整体输出不会发生剧烈偏差。FSM 在 CO₂ 传感器选型上刻意避开了 PIR 红外方案——PIR 仅对运动目标敏感，无法检测静坐阅读的人员，在图书馆场景中存在致命的功能盲区；CO₂ 则不受此限制，人员只要在呼吸就会持续产生信号。

引擎二为 LLM 智能顾问，仅在 FSM 判定为 ARRIVING 或 OCCUPIED 状态时按需激活。LLM 接收一个结构化的 JSON 输入，包含当前全部传感器的实时读数、FSM 状态与态势评分值、当前生效的用户偏好模式配置（节能优先、均衡模式或舒适优先）以及来自 OpenWeather API 的室外天气数据（温度、湿度、天气状况和风速）。系统将这些异构信息组装为标准化 Prompt 后发送至 DeepSeek API，要求 LLM 以严格的 JSON Schema 返回三项结构化输出——照明亮度建议（0 至 100 的整数）、风扇启停建议（布尔值）和不超过 200 字的中文推理说明。LLM 返回的 JSON 经本地照度保护逻辑校验：若当前实际照度已高于当前模式的补光阈值，即使 LLM 建议开启照明，本地校验也会将亮度强制置零并附加"后端照度保护"标记写入推理文本。LLM 不可用时（API 密钥未配置、网络超时或连续三次调用失败），系统自动切换至 FSM Fallback 模式，由本地规则引擎根据当前传感器读数和用户偏好模式独立生成控制决策和推理说明，控制功能在 LLM 降级期间完全不中断。

### 2.2 安全告警双通道并行执行机制

在典型的智能物联网系统设计中，安全告警功能通常作为 AI 决策引擎的一个子模块运行——告警的触发依赖于 AI 链路的状态判定结果，告警指令也经由 AI 链路的通信通道下发。这种设计的隐患在于：AI 链路本身成为了安全告警的单一故障点——大语言模型服务不可用、FSM 状态机线程异常退出、PC 端与 JACE8000 的网络连接中断，任一环节的失效都可能导致烟雾超标时蜂鸣器无法触发。

本系统将安全告警功能从 AI 决策链路中彻底剥离，为其构建了独立的双通道并行执行架构。在 JACE8000 本地，Niagara Wire Sheet 中预置了不依赖任何外部通信的告警规则——直接读取 IO-22U 的烟雾和噪声通道寄存器值，在满足触发条件时通过 DO 通道直接驱动蜂鸣器和警示 LED，整个链路从传感器采样到执行器动作全部在 JACE8000 内部完成，延迟在 50 毫秒以内。与此同时，PC 端 Python 后端的 DeviceController 模块在每个控制循环（2 秒周期）中独立执行相同的告警规则，通过 oBIX 协议读取传感器值并写入执行器指令，同时将告警事件记录至 SQLite 数据库并推送给前端 Dashboard 触发弹窗和横幅联动。两条通道各自独立运行、互不依赖——JACE8000 通道不依赖 PC 端的任何服务，PC 端通道也不依赖 JACE8000 Wire Sheet 的告警输出——任一通道的失效不影响另一通道的告警功能。

告警规则本身引入了完整的回滞区间和持续时间判据，以保证告警输出的可靠性。烟雾告警：烟雾浓度 ≥ 150 ppm 时立即触发蜂鸣器，浓度回落至 130 ppm 以下并持续 5 秒后方可解除，20 ppm 的回滞区间防止浓度在阈值附近波动导致的反复触发。噪声告警：噪声 ≥ 65 dB 且持续超过 10 秒时触发警示 LED，噪声降至 58 dB 以下并持续超过 20 秒后方可解除，7 dB 的回滞和两级持续时间判据分别用于防止瞬时噪声误报和告警状态闪烁。安全告警双重执行流程示意如下：

### 2.3 三层递进式降级与自治恢复机制

在物联网实时控制系统中，从云端 AI 服务到边缘控制器之间的任何单一环节失效——网络抖动导致的 API 超时、PC 端进程的异常退出、JACE8000 与 IO-22U 之间的 RS-485 通信中断——都可能导致整个控制链路的静默崩溃。更关键的是，上层功能的失效不应向下传导：AI 不可用不应导致基础规则控制中断，PC 端离线不应导致安全告警失效。针对这一需求，本系统设计了三层递进式的降级与自治恢复体系。

第一层为完整功能层。LLM 顾问与 FSM 状态机在 PC 端协同运行，本地规则引擎综合 FSM 状态、偏好模式配置和传感器阈值独立计算控制指令，通过 oBIX 协议下发至 JACE8000。安全告警由 PC 端规则引擎和 JACE8000 本地 Wire Sheet 同步执行。

第二层为 LLM 降级层。当 LLM 服务不可用——API 密钥未配置、网络超时或连续三次调用返回异常——系统自动将 AI 模式标记为"FSM Fallback"，由本地规则引擎根据传感器读数和偏好模式阈值自主决策。切换过程对外部控制功能完全透明：照明和风扇的自动控制不会出现中断或抖动，Dashboard 上的 AI 状态指示灯从绿色切换为黄色以提示运维人员。

第三层为边缘自治层。当 PC 端与 JACE8000 之间的 oBIX 通信中断时，PC 端的控制循环自然停止向 JACE8000 写入新的控制指令。此时 JACE8000 内置的 Niagara Wire Sheet 图形化逻辑自动接管——Wire Sheet 中预置了基于传感器阈值的基础控制规则（温度高于阈值时开启风扇、光照低于阈值时开启照明、烟雾超标时触发蜂鸣器），这些规则直接运行在 JACE8000 的 QNX 实时操作系统上，不依赖 PC 端的任何服务。当 PC 端通信恢复后，BackendRuntime 重新建立 oBIX 连接并接管控制权，Wire Sheet 自动退居为纯保底角色。从降级触发到完全恢复的整个过程无需人工干预。

尤为关键的是，烟雾→蜂鸣器和噪声→警示LED这两条安全告警规则在本体系中具有特殊地位——它们在三个层级中各自独立实现，不参与降级。即使 LLM 顾问和 FSM 状态机同时失效，JACE8000 仍能在检测到烟雾超标的毫秒级时间内通过本地 Wire Sheet 触发蜂鸣器。三层降级保底切换逻辑示意如下：

前端交互层是用户与系统之间的操作界面，以 B/S 架构的 Web Dashboard 形式运行于浏览器端。用户无需安装任何客户端软件，通过 PC、平板或手机浏览器即可访问全部功能。该层围绕实时监控、设备管理、数据分析和智能交互四条主线展开，具体包含以下功能：

1. 多页面导航。Dashboard 采用侧边栏导航结构，包含主控看板、AI 助手、3D 模型、历史趋势和能耗分析五个页面。页面间切换通过前端路由实现，无需刷新整个页面，当前所在页面在侧边栏以高亮色条标识。

2. 实时数据呈现。主控看板以 2×5 卡片网格布局展示全部 9 项传感器数据（温度、湿度、光照度、CO₂、噪声、烟雾、PM2.5、风扇电流、风扇功率），每张卡片包含传感器图标、实时数值、单位和色彩健康度指示——绿色表示数值处于正常范围，黄色表示接近阈值边界，红色表示已超出安全或舒适阈值。数据通过 AJAX 每 2 秒向后端 API 发起轮询并局部刷新卡片，页面其余部分不重新渲染。

3. 设备状态查看与控制。设备控制面板以卡片形式展示 4 个执行器的当前状态：蜂鸣器和警示 LED 显示 ON/OFF 状态及触发原因，照明 LED 显示亮度百分比并附带拖动滑块，风扇显示启停状态。每张设备卡片设有自动/手动模式切换开关，手动模式下用户可直接开关蜂鸣器、调节照明亮度或启停风扇，操作结果通过 Toast 通知即时反馈。

4. 告警推送与联动。烟雾告警触发时，主控看板正中央弹出模态遮罩弹窗，显示告警类型、当前烟雾浓度值、触发阈值和简要处置指引，用户需点击"我已知晓"按钮关闭弹窗。噪声告警触发时，页面底部显示红色滚动横幅，标注噪声分贝值和持续时间。告警解除后弹窗自动消失、横幅自动隐藏，告警的触发与解除事件均记录至数据库的可追溯。

5. 历史趋势分析。历史趋势页左侧为传感器复选框选择面板，右侧为 Canvas 绘制的多轴折线图。用户可同时勾选多个传感器，每个传感器对应一条不同颜色的折线，通过顶部预设按钮在 1 小时、6 小时、24 小时和 7 天四个时间范围间切换。图表上方工具栏提供 CSV 数据导出和 PNG 图表导出功能。

6. AI 对话助手。AI 助手页面采用聊天气泡式界面，左侧为对话历史展示区，底部为文本输入框和语音按钮。系统预置四个快捷提问按钮——"分析当前环境数据""查看近期趋势""查询应急预案""分析当前能耗"——用户点击即可自动填入提问内容。AI 回复以模拟流式输出的方式逐段显示。每次对话自动附带当前传感器读数、FSM 状态和活跃告警列表作为上下文信息，确保 AI 回答基于实时数据而非过时信息。

7. 3D 可视化模型。基于 Three.js 构建的图书馆三维场景通过 iframe 嵌入 Dashboard 的 3D 模型页面。场景中标注了传感器安装位置和执行器部署点位，传感器图标颜色随实时数据动态变化，执行器图标在设备动作时呈现动画效果（风扇叶片旋转、照明灯发光等）。用户可通过鼠标旋转、缩放和平移视角，从任意角度观察图书馆的设备布局。该模块接收主控看板的设备控制指令并同步更新三维场景中的设备状态。

#### 3.1.2 后端服务

后端服务层是系统的数据和计算中枢，部署于 PC 端 Python 运行环境中。它承接边缘控制层上传的传感器数据，驱动 FSM 状态机和 LLM 顾问完成智能决策，并通过 RESTful API 向前端提供全部数据接口。该层的核心职能可归纳为以下五个模块：

1. 实时数据获取。系统通过 ObixClient 模块与 JACE8000 的 oBIX 服务建立持久化 HTTPS 连接（启用 Session 复用和 Basic Auth 认证），以 2 秒为周期向全部 8 个传感器点位依次发起 HTTP GET 请求，解析返回的 XML 响应提取数值。每个传感器独立追踪在线状态——读取成功则标记 online 并更新时间戳，读取失败（超时、连接拒绝或 XML 解析异常）则标记 offline 并记录错误信息，同时自动使用上一次有效读数填充当前值以保证控制循环的连续性。数据获取层支持在部分传感器离线时继续正常运行，并在全部传感器恢复后自动切换回完整数据模式。

2. 数据存储与管理。系统采用 SQLite 数据库作为本地数据存储方案，数据库文件位于后端程序同级目录。五张核心数据表分别存储传感器时序数据（sensor_readings）、设备操作事件日志（device_events）、FSM 状态迁移记录（fsm_state_log）、LLM 决策记录（llm_decisions）和告警生命周期记录（alerts）。数据库启用 WAL 模式以支持后台写入线程与前台 API 查询线程的并发访问，sensor_readings 表建立 (sensor_name, timestamp) 复合索引以加速按传感器和时间范围查询。系统按 7 天保留周期自动清理过期数据，防止数据库文件无限增长。

3. API 接口服务。基于 Flask 框架的 Web 服务器以应用工厂模式构建，接收 BackendRuntime 实例作为依赖注入。API 基础路径为 /api/v1，共提供 17 个 RESTful 端点，覆盖传感器数据查询（latest/history/diagnostics）、设备控制（status/control/events/diagnostics）、系统管理（status/profiles/profile）、天气数据、能耗统计（summary/timeseries）、告警查询、AI 对话助手（chat/history/quick-prompts）等功能域。所有端点返回 JSON 格式数据，配置 ensure_ascii=False 以原生支持中文输出。前端静态文件通过 Flask 的 send_from_directory 从 frontend/ 目录对外服务，统一挂载在 /app/ 路径下。

4. AI 智能分析服务。系统集成两个外部 AI 能力：DeepSeek API（deepseek-chat 模型）作为 LLM 顾问引擎的后端推理服务，接收结构化 JSON Prompt 并返回包含照明亮度建议、风扇启停判断和中文推理说明的 JSON 输出，经本地 Pydantic 风格字段校验后传递给规则引擎；OpenWeather API 以 10 分钟 TTL 缓存获取武汉市室外天气数据（温度、体感温度、湿度、风速、天气描述），作为 LLM 决策的多模态参考输入。AI 对话助手模块独立于 LLM 顾问引擎运行，采用基于关键词匹配和上下文规则的自然语言问答路由机制，根据用户提问内容自动分发至环境分析、趋势分析、能耗分析或应急预案四类回答模板。

5. 智能决策引擎。BackendRuntime 类作为后端的总调度中心，在独立守护线程中运行主控制循环（周期 2 秒）。每轮循环依次执行：调用 SensorDataCollector 采集最新传感器读数 → 调用 OccupancyDetector 计算综合态势值 S_t 并驱动 FSM 状态迁移 → 调用 DeviceController 执行安全规则（烟雾/噪声判据）和舒适性控制（照明/风扇指令）→ 将本轮数据存入数据库。LLM 顾问在 FSM 为非 VACANT 状态时异步激活，其输出不阻塞主循环。

#### 3.1.3 边缘控制

边缘控制层是系统与物理环境直接交互的接口，以 JACE8000 边缘控制器为核心，通过 IO-22U 扩展 I/O 模块接入全部传感器和执行器。该层在承担数据采集与指令执行的基础职能之外，还具备独立的本地自治能力——当上层服务不可用时，JACE8000 内置的 Wire Sheet 逻辑可接管控制权，确保安全告警等关键功能持续运行。该层包含以下四个功能模块：

1. 传感器数据采集。IO-22U 的 8 路通用输入通道（UI-1 至 UI-8）全部配置为 4-20mA 电流模式，分别接入温湿度变送器（温度+湿度两路）、光照传感器、CO₂ 传感器、噪声传感器、烟雾传感器、PM2.5 传感器和电流变送器。IO-22U 内置的 12 位可编程增益放大器（PGA）配合模数转换器（ADC）将模拟电流信号转换为数字量，精度为 ±0.01mA。转换后的数据通过 RS-485 物理层以 Modbus RTU 协议上传至 JACE8000（IO-22U 作为从站地址 1，JACE8000 作为主站轮询，通信参数为 38400 bps、8 位数据位、无校验、1 位停止位）。每路传感器在 Niagara Workbench 中配置 Linear Conversion 扩展，将 4-20mA 信号线性映射至对应工程单位量程。

2. 执行器设备控制。IO-22U 的 3 路数字输出通道（DO-1 至 DO-3）以继电器触点形式控制蜂鸣器、警示 LED 和风扇的通断，继电器规格为 SPST NO、24VAC/DC 1A。1 路模拟输出通道（AO-1）以 0-10V 电压信号控制照明 LED 的亮度，12 位 DAC 提供 4096 级分辨率，实现 0-100% 的无级调光。执行器可接收来自两条通路的控制指令：正常运行时由 PC 端通过 oBIX 协议下发指令，PC 端离线时由 JACE8000 本地 Wire Sheet 接管控制权。

3. 硬件状态监测。IO-22U 每个通道独立追踪在线状态，通过 Modbus 通信超时检测和信号范围校验识别传感器离线或故障——4-20mA 信号低于 3.6mA 或高于 20.4mA 时即判定为异常。单个通道的故障不影响其余通道的正常数据采集和控制输出，PC 端后端的 SensorDataCollector 模块同步维护各通道的在线/离线状态表。

4. 本地自治控制。JACE8000 内置的 Niagara Wire Sheet 图形化编程环境中预置了两组关键控制逻辑：一是基于传感器阈值的设备控制规则（温度高于当前模式阈值时闭合风扇继电器、光照低于阈值时输出照明控制电压），确保基础环境调控不依赖 PC 端；二是安全告警规则（烟雾浓度 ≥ 150ppm 时闭合蜂鸣器继电器、噪声 ≥ 65dB 时闭合警示 LED 继电器），确保安全功能在极端断网场景下仍然有效。Wire Sheet 逻辑运行于 JACE8000 的 QNX 实时操作系统上，从传感器采样到执行器动作的全链路延迟在 50 毫秒以内。当 oBIX 通信恢复后，PC 端 BackendRuntime 自动重新建立连接并接管控制权，Wire Sheet 退居为纯保底角色。

### 3.3 智能决策控制功能

智能决策控制功能是后端服务层的核心，由 BackendRuntime 主控制循环统一调度 OccupancyDetector、OccupancyFSM、LLMService 和 DeviceController 四个子模块协同工作。该功能遵循"态势量化→状态迁移→LLM 顾问→指令执行"四步流水线，每 2 秒完成一轮完整的感知-决策-控制闭环。以下按流水线顺序逐一阐述各步骤的设计。

> **[此处留图：AI 决策四步流水线图]**
>
> 纵向四段流水线布局，四个阶段用不同颜色横条区分，数据从上向下流动。第一段（蓝色横条）：态势量化——左侧为五个传感器小球（CO₂/噪声/温度Δ/湿度Δ/光照Δ，五种颜色）汇聚到一个加权求和符号 Σ，符号内部标注"40%+25%+10%+10%+15%"，右侧输出为圆角矩形"综合态势值 S_t ∈ [0,1]"，矩形颜色随数值从灰渐变到绿。第二段（橙色横条）：状态迁移——左侧输入 S_t 数值和当前状态，中间为四态简化状态图（四个小圆球 V/A/O/L 纵向排列，圆球间用弧形箭头连接，当前激活态高亮放大），右侧输出为当前 FSM 状态标签和 LLM 激活标志（绿色勾号=激活、灰色叉号=静默）。第三段（绿色横条）：LLM 顾问——左侧三路输入汇入（FSM 状态、传感器 JSON 数据块图标、天气云朵图标），中间为 DeepSeek API 云图标（标注"temperature=0.1, max_tokens=300"），右侧输出为结构化 ControlDecision 对象（灯泡图标+风扇图标+文字气泡图标，用虚线框标记"仅供参考"黄色标签）。第四段（紫色横条）：指令执行——左侧两路汇入（LLM 建议虚线箭头+FSM 规则实线箭头），中间为齿轮图标（规则引擎），右侧分叉为两路输出：上路 AO-1 模拟量→照明 LED 图标（旁注亮度百分比），下路 DO-3 开关量→风扇图标（旁注 ON/OFF），两路之间为一道红色虚线表示安全旁路，从流水线左侧传感器层直接跨越到右侧蜂鸣器和警示 LED 图标，标注"安全告警独立旁路"。

#### 3.3.1 态势量化

态势量化是整个决策链路的起点，由 OccupancyDetector 模块负责。该模块读取最新的全部传感器快照数据后，将当前采样值与上一轮快照比较，计算温度变化量、湿度变化量和光照变化量的绝对值作为派生特征——这三个变化量反映的是人员活动引起的环境微变，而非传感器读数的绝对值，因此比直接使用温度或光照的原始值更具空间占用指示力。

随后，五个特征分别按预设量程进行归一化处理，经线性映射压缩至 [0, 1] 区间：CO₂ 映射范围为 [450, 1200] ppm——450 ppm 为室外大气 CO₂ 基准浓度，低于此值视为无人员信号，1200 ppm 以上视为满信号；噪声映射范围为 [30, 70] dB——30 dB 以下为极端安静环境，70 dB 以上为人员活动密集的强信号；温度变化率映射范围为 [0, 1.5] °C/采样周期——反映人员体热引起的局部温度微升；湿度变化率映射范围为 [0, 8.0] %RH/采样周期——反映人员呼吸和体表蒸发引起的湿度微增；光照变化率映射范围为 [0, 350] Lux/采样周期——反映人员走动遮挡光线或操作窗帘引起的照度波动。

各特征归一化后以预设权重加权求和，得到综合态势值 S_t。权重分配以各传感器与人员存在之间的物理因果可靠性为依据：CO₂ 因人体新陈代谢持续产生且气体扩散具有物理惰性——浓度不会因人员短暂离开而瞬间下降——作为最可靠的人员存在间接指标，权重最高为 40%；噪声反映人员交谈和活动水平，在图书馆场景中与人员存在具有较强关联，权重为 25%；温度变化率、湿度变化率和光照变化率三个派生特征分别表征人员体热、呼吸湿气和行为对环境造成的微扰动，权重依次为 10%、10% 和 15%，合计 35%。烟雾传感器和 PM2.5 传感器不参与态势量化——前者是安全信号，不应影响舒适性决策；后者颗粒物浓度与人员存在之间的因果关系较弱，纳入加权可能引入噪声干扰。

#### 3.3.2 状态迁移

OccupancyFSM 根据态势值 S_t 执行四态状态机迁移。系统定义了四种空间占用状态：VACANT（空闲）——空间判断为无人，照明和风扇强制关闭以最大限度节能；ARRIVING（到达）——态势值从 VACANT 的极低水平攀升至 0.40，传感器开始感知到人员活动迹象，系统进入过渡预备状态；OCCUPIED（占用）——态势值突破 0.45 阈值，系统确认空间处于有人状态，LLM 顾问激活并开始提供环境优化建议；LEAVING（离开）——态势值从 OCCUPIED 高位回落至 0.40 以下，系统判定人员开始离开，进入过渡退出状态。

状态迁移采用非对称滞回策略，利用 CO₂ 浓度的物理惯性抑制频繁翻转。进入 OCCUPIED 的门槛为 S_t ≥ 0.45，但退出 OCCUPIED 回到 VACANT 的最终门槛——经 LEAVING 过渡阶段后——仅需 S_t ≤ 0.30。0.15 的滞回差距意味着：人员短暂离开阅览区去取书或接水时，CO₂ 浓度不会瞬间下降，S_t 仍停留在滞回区间内，系统保持当前状态不变；只有人员长期离开、CO₂ 浓度持续下降后才会确认退出占用状态。这一机制天然抑制了传感器噪声和人员短暂移动引发的状态反复翻转。

各状态间的具体迁移规则如下：VACANT 态下，S_t ≥ 0.40 迁移至 ARRIVING，S_t ≥ 0.45 直接迁移至 OCCUPIED——对应快速来人时跳过过渡状态的场景；ARRIVING 态下，S_t ≥ 0.45 迁移至 OCCUPIED，S_t 回落至 0.20 以下退回 VACANT——排除传感器瞬时扰动引发的误判；OCCUPIED 态下，S_t < 0.40 迁移至 LEAVING；LEAVING 态下，S_t ≤ 0.30 迁移至 VACANT——确认人员已离开，S_t 重新回升至 0.45 则回到 OCCUPIED——对应人员中途返回的场景。状态每 2 秒更新一次，与传感器采集周期保持同步。

#### 3.3.3 LLM 顾问

LLM 顾问引擎是双引擎架构中的智能层，由 LLMService 模块实现。引擎仅在 ARRIVING 或 OCCUPIED 状态下激活——VACANT 状态下不调用 LLM，直接由 FSM 本地规则输出关闭指令以节省 API 调用开销；LEAVING 状态下同样不调用 LLM，由本地规则逐步降低设备输出。

激活时，LLMService 构造结构化的 JSON 输入：system prompt 明确限定 LLM 的角色为"图书馆物联网舒适性控制助手"，仅负责照明亮度建议和风扇启停建议，不得处理烟雾和噪声等安全动作；user message 以 JSON 格式传入全部传感器实时读数、当前 FSM 状态、偏好模式配置参数（温度阈值和照明阈值）、来自 OpenWeather API 的室外天气数据（天气状况与室外温度），以及 CO₂ 舒适上限 1200 ppm 等规则约束。请求以 temperature=0.1 的低随机性、max_tokens=300 的紧凑输出和 response_format=json_object 的结构化约束发送至 DeepSeek API，超时时间设为 8 秒。

LLM 返回的 JSON 对象——包含 lighting_brightness（0-100 整数）、fan_state（布尔值）和 reasoning（中文简短理由）——经正则表达式兜底提取 JSON 片段后进行字段完整性校验。校验通过后，输出还必须经过本地照度保护逻辑：若 LLM 建议的照明亮度大于零，而当前实际光照度已高于当前偏好模式的补光阈值，照明建议将被强制置零，同时在推理说明中追加"后端照度保护"提示信息。这一保护机制确保 LLM 即使在不了解当前实际照度的情况下给出不合理的高亮度建议，也不会造成照明能耗的浪费。风扇建议不设本地保护——温度与 CO₂ 双参数的最终判别在 DeviceController 的规则引擎中完成。

当 LLM_API_KEY 未配置、LLM_ENABLED 为 false、API 调用超时或返回格式异常时，LLMService 自动切换至 FSM Fallback 模式——由 _fallback_decide 方法根据当前偏好模式的温度阈值和光照阈值，直接输出基于本地规则的控制建议，确保 LLM 不可用时系统控制的连续性和确定性。

#### 3.3.4 指令执行

指令执行是决策链路的最后一步，由 DeviceController 的 _apply_comfort_control 方法统一完成。该方法接收 FSM 当前状态、偏好模式配置参数、传感器实时快照和 LLMService 返回的 ControlDecision 对象，综合计算照明亮度和风扇状态的最终控制值。

照明控制方面，若 FSM 状态为 VACANT，亮度强制置零；否则读取偏好模式中的 light_on_below_lux 阈值，与当前光照度比较——实际照度高于阈值时亮度置零，低于阈值时输出偏好模式预设的 lighting_brightness 值（节能 50%、均衡 60%、舒适 80%）。风扇控制方面，以温度和 CO₂ 浓度为双参数判据，综合偏好模式中的 fan_on_above_c 阈值，通过 _guard_fan_state 方法（含 1°C 温度回滞）计算最终启停指令。

控制指令确定后，DeviceController 调用 _set_lighting 和 _set_bool_device 方法，比较目标状态与缓存的当前状态，仅在状态确实发生变化时才通过 oBIX 协议向 JACE8000 对应点位写入控制指令，避免冗余通信加重总线负载。照明 LED 的亮度百分比经线性映射（brightness% ÷ 100 × 10V）转换为 0 至 10V 模拟电压值，经 IO-22U 的 AO-1 通道输出；风扇继电器通过 DO-3 通道的触点通断驱动。每次控制指令发生变化时，决策签名——亮度值、风扇状态、FSM 状态三元组——被记录至 SQLite 数据库的 llm_decisions 表，用于后续能耗统计和决策回溯分析。

安全告警规则——烟雾浓度超标触发蜂鸣器、噪声持续超标触发警示 LED——通过 DeviceController 的 _apply_safety_rules 方法在指令执行步骤中独立并行运行。安全规则的判断逻辑和输出通道与舒适性控制通道完全隔离，不经过态势量化和 FSM 决策链路的任何环节，确保安全响应具有零延迟和绝对的确定性。

### 3.4 设备自适应控制功能

设备自适应控制功能运行于后端服务层的规则引擎中，接收 FSM 状态和 LLM 建议作为输入，向边缘控制层的 IO-22U 输出通道下发最终控制指令。控制指令以 oBIX XML 格式经 HTTPS 发送至 JACE8000，由 JACE8000 中转至 IO-22U 的 DO/AO 通道驱动执行器动作。在整个控制链路中，LLM 的建议仅作为参考输入——本地规则引擎根据传感器实测阈值和偏好模式配置进行最终的确定性决策，LLM 在任何情况下均不直接控制硬件。控制范围覆盖照明 LED 和风扇两台执行器，以温度、CO₂ 浓度和光照度为主要环境判据，同时遵循空间状态优先原则——VACANT 状态下照明和风扇无论传感器读数如何均强制关闭，从架构层面杜绝无人时段的无谓能耗。以下按照明控制、风扇控制和偏好模式三个子功能逐一展开。

#### 3.4.1 照明控制

照明控制以光照度与补光阈值的比较为核心逻辑，执行一条简洁的条件判断：当 FSM 状态非 VACANT 时，读取当前光照度传感器读数，与当前偏好模式的补光阈值比较——若实际照度 ≥ 阈值，说明自然光充足，照明亮度置零；若实际照度 < 阈值，说明自然光不足，照明输出偏好模式预设的亮度百分比值。VACANT 状态下无论光照条件如何，照明亮度强制为零。

照明 LED 通过 IO-22U 的 AO-1 模拟输出通道驱动。AO-1 支持 0-10V 电压输出，DeviceController 将 0-100% 的亮度百分比线性映射为 0-10V 的模拟电压值——例如均衡模式下 60% 的亮度对应 6.0V 输出——经 IO-22U 内置的 12 位数模转换器（DAC）转换为模拟电压信号，驱动 LED 驱动器实现无级调光。模拟调光相比 PWM 数字调光具有无频闪、调光平滑、适配通用 LED 驱动器的优势，更适合图书馆阅览区对视觉舒适度有较高要求的场景。照明 LED 支持手动覆盖模式——管理员可通过 Dashboard 的亮度滑条或 POST /api/v1/devices/control API 接口将亮度直接设为 0-100% 之间的任意值，手动模式下自动决策暂停，切换回自动模式后恢复。

#### 3.4.2 风扇控制

风扇控制引入温度-CO₂ 双参数联合判据，避免单一温度阈值在过渡季节或 CO₂ 积聚场景下出现控制盲区。开启条件为"温度超过偏好模式风扇开启阈值"或"CO₂ 浓度超过 1200 ppm"二者满足其一——前者对应夏季高温需降温通风的典型场景，后者对应冬季门窗紧闭、CO₂ 积聚但室温不高时同样需要启动风扇换气的场景。关闭条件为两个参数同时满足各自的回落阈值：温度降至模式阈值以下 1°C 且 CO₂ 不超过 900 ppm，二者缺一不可。

系统通过 1°C 温度回滞和 300 ppm CO₂ 回滞两个滞回区间防止风扇在阈值边界频繁启停。回滞的实际效果为：温度在开启阈值 ± 1°C 范围内小幅波动时，风扇维持当前状态不变，避免短周期反复启停损伤设备寿命；CO₂ 在 900-1200 ppm 区间内波动时同样维持当前状态，仅在明确回落至 900 ppm 以下且温度也满足关闭条件时才允许停机。当前状态处于回滞区间内时，系统维持上次的启停状态不变。风扇启停通过 IO-22U 的 DO-3 继电器通道控制——继电器闭合时风扇回路导通（开启），断开时回路切断（关闭）。

风扇的实时功率由串联在 24V DC 供电回路中的电流变送器间接测量。变送器采用非侵入式接线方式——变送器的 IN+/IN- 端子串联在 24V 电源与风扇之间，V+/V- 端子由 24V 电源独立供电，OUT+/OUT- 端子输出 4-20mA 电流信号至 IO-22U 的 UI-8 通用输入通道。经 IO-22U 内置 12 位 ADC 转换和 Niagara 线性换算后，采集程序按 P = 24V × I 关系式将电流值换算为功率瓦数。非侵入式测量方式无需在风扇回路中串接额外的功率表或分流电阻，接线简洁且不影响风扇的正常运行。风扇同样支持手动覆盖——管理员可通过 Dashboard 或 API 强制开启或关闭风扇，手动模式下自动决策暂停，切换回自动模式后恢复。

#### 3.4.3 用户偏好模式

系统提供节能优先、均衡模式和舒适优先三种用户偏好模式，通过调整风扇开启温度阈值和照明亮度参数，使同一套控制逻辑能够适配图书馆在不同时段和不同运营场景下的差异化需求。管理员可通过前端 Dashboard 侧边栏的模式切换器或 POST /api/v1/system/profile API 接口实时切换模式，切换后下一个 2 秒控制周期即生效，无需重启系统或中断当前服务。

三种模式的核心差异在于"节能-舒适"的权衡取舍，参数配置如表 3-2 所示。节能优先面向寒暑假、夜间闭馆后等低人流时段，风扇开启温度最高（30°C），照明补光阈值最低（100 Lux），亮度最暗（50%），以最小化能耗为首要目标——温度即使偏高也不轻易开启风扇，自然光即使偏暗也不轻易开灯。舒适优先面向考试季和学术活动等人流密集且环境品质要求高的场景，风扇开启温度最低（26.5°C），照明补光阈值最高（200 Lux），亮度最高（80%），以保证人体舒适度为优先目标——温度略有升高即开启风扇，自然光稍有不足即补光。均衡模式作为出厂默认配置，参数介于两者之间，适合日常运营的多数时段。

### 3.5 安全告警管理功能

安全告警管理功能是唯一在边缘控制层和后端服务层中双重独立实现的功能模块——JACE8000 本地 Wire Sheet 和 PC 端 DeviceController 各自运行完全独立的告警规则，互不依赖。这种双通道并行架构是系统安全设计的基本保证：即使 PC 端 AI 引擎故障、oBIX 通信中断或 Flask 服务完全宕机，JACE8000 内置的 Wire Sheet 逻辑仍然能够在边缘侧独立检测烟雾和噪声异常并驱动蜂鸣器与警示 LED，确保"断网不断控、AI 失效保底仍在"的安全底线。PC 端则在此基础上提供告警事件的数据库持久化、Dashboard 弹窗联动和运维追溯等上层增值能力。

>
> 烟雾告警以烟雾浓度为核心判据，警报阈值为 150 ppm，解除阈值为 130 ppm，中间 20 ppm 的回滞区间防止传感器在阈值边界微小抖动时引发蜂鸣器频繁通断。烟雾浓度一旦触及 150 ppm，蜂鸣器立即无条件触发——不经过态势量化和 FSM 决策链路的任何环节，延迟控制在单个控制周期（2 秒）以内。浓度回落至 130 ppm 以下后，系统启动 5 秒持续确认计时器：只有当浓度在连续 5 秒内始终低于 130 ppm 时，蜂鸣器才解除，避免烟雾浓度的瞬时波动导致告警提前解除。触发同时，PC 端 DatabaseManager 以 open_alert 方法向 alerts 表写入一条 severity=critical 的告警记录，包含触发时刻、当前烟雾浓度实测值和触发阈值，供事后追溯分析。蜂鸣器通过 IO-22U 的 DO-1 继电器通道驱动，继电器闭合时蜂鸣器回路导通发声，断开时静音。

噪声告警相比烟雾告警引入了更复杂的时序判据，以防止短促的瞬时噪声（如书本掉落、椅子拖动）误触发告警。判断流程分为上升沿和下降沿两个方向。上升沿方向：噪声值首次超过 65 dB 阈值时启动 10 秒持续计时器，只有当噪声在连续 10 秒内始终维持在 65 dB 以上时，才判定为确认的噪声超标事件并触发警示 LED——10 秒的持续时间有效过滤了短暂噪声尖峰。下降沿方向：噪声值回落至 58 dB（65 dB − 7 dB 回滞）以下时启动 20 秒持续确认计时器，只有当噪声在连续 20 秒内始终低于 58 dB 时，警示 LED 才熄灭——20 秒的确认时间远长于上升沿的 10 秒触发时间，避免在噪声源间歇性发作时警示 LED 频繁闪烁。这种"短触发、长解除"的非对称时序设计确保了告警状态的稳定性和可观察性。噪声告警以 severity=warning 级别写入 alerts 表，警示 LED 通过 IO-22U 的 DO-2 继电器通道驱动。

在 PC 端，安全告警规则集中实现于 DeviceController 的 _apply_safety_rules 方法中。该方法在每轮控制循环中首先被执行，独立于 _apply_comfort_control 舒适性控制方法，两者使用完全独立的判断变量和输出通道——安全规则写入 DO-1 和 DO-2，舒适性控制写入 AO-1 和 DO-3。告警触发时，前端 Dashboard 通过 /api/v1/alerts 接口实时获取活跃告警列表，联动触发红色告警弹窗和底部滚动告警横幅。告警弹窗包含告警类型、严重等级、触发数值和触发时间四项信息，底部横幅持续滚动显示所有活跃告警的摘要文字。当存在活跃告警时，DeviceController 的 ai_mode 字段自动切换为 emergency_override，此时 LLM 顾问和 FSM 舒适性决策均被暂停，系统进入纯安全响应模式，舒适性设备（照明 LED 和风扇）保持上一次的正常状态不变，直至全部告警解除后自动恢复。告警事件的完整生命周期——从触发到解除的时间戳、传感器实测值、触发阈值——均在 SQLite alerts 表的 triggered_at 和 resolved_at 字段中持久记录，支持按时间范围和告警类型进行历史查询。

## 第四章  系统实现

本章按照物联网系统的经典四层架构——感知层、传输层、控制层和应用层——自底向上阐述系统的具体实现方案，最后以配置参数速查表收尾。感知层对应硬件设备的选型、接线和通道分配，解决"数据从哪里来、指令到哪里去"的物理基础问题；传输层对应 oBIX 协议通信链路，解决 JACE8000 与 PC 端之间的数据交换问题；控制层对应 Python 后端各模块的实现逻辑，解决"如何从传感器数据推导出控制指令"的智能决策问题；应用层对应 Flask Web 服务、前端 Dashboard 和数据库设计，解决"用户如何看到数据、操作系统"的交互问题。以下按层次逐一展开。

### 4.1 物联网感知层

物联网感知层是系统与物理世界交互的最底层，负责完成环境数据的采集、模数转换和执行器控制信号的输出。本系统的感知层采用"JACE8000 边缘控制器 + VYKON IO-22U 扩展 I/O 模块"的双级架构——JACE8000 作为主控制器运行 Niagara Framework 4 并对外提供 oBIX 服务，IO-22U 作为从站通过 RS-485 Modbus RTU 总线与 JACE8000 通信，负责接入全部传感器和执行器。传感器全部采用工业级 4-20mA 电流信号变送器，相比普通数字传感器具有抗干扰能力强、传输距离远、自带断线诊断（活零）等优势。以下依次介绍 JACE8000 和 IO-22U 的技术参数、通道分配方案以及电流变送器的接线方式。

#### 4.1.1 JACE8000 边缘控制器

JACE8000 是霍尼韦尔 Tridium 推出的嵌入式物联网控制引擎及服务器平台，搭载 TI AM3352 1000MHz ARM Cortex-A8 处理器、1GB DDR3 SDRAM 内存、4GB Flash 闪存及 2GB Micro SD 用户存储空间，配备双 10/100MB 以太网口和双隔离 RS-485 接口，支持 802.11a/b/g/n Wi-Fi 无线连接，运行 QNX 实时操作系统和 Niagara Framework 4 软件平台。

JACE8000 在本系统中作为边缘控制核心，负责通过 IO-22U 模块采集传感器数据、执行控制指令，并通过 oBIX 协议将数据暴露给上层 PC AI 引擎。

#### 4.1.2 VYKON IO-22U 扩展 I/O 模块

VYKON IO-22U 是一款支持 Modbus RTU 和 BACnet MSTP 双协议的通用 I/O 模块，搭载 ARM Cortex 32-bit 24MHz 处理器。其 I/O 资源配置如下：8 路通用输入通道支持 0-10V、0-5V、4-20mA、0-20mA 电压/电流信号以及电阻、NTC 和 RTD 温度传感器输入，内置 12 位 ADC 和可编程增益放大器；4 路数字输入通道支持干接点和 NTC 10K 热敏电阻接入；8 路数字输出通道为 SPST NO 型继电器，触点容量 24VAC/DC 1A；2 路模拟输出通道支持 0-10V 电压和 4-20mA 电流输出，内置 12 位 DAC；1 路 RS-485 通信接口，作为 Modbus RTU 从站运行，默认波特率 38400 bps。

#### 4.1.3 硬件模块介绍

本系统涉及的物理硬件分为传感器输入设备和执行器输出设备两大类，共计 8 个传感器和 4 个执行器。所有传感器均采用工业级 4-20mA 电流信号变送器，相比普通数字传感器具有抗干扰能力强、传输距离远、自带断线诊断（活零状态下输出 4mA，断路时电流归零可被 IO-22U 立即检测）等优势。执行器通过 IO-22U 的 DO 继电器通道或 AO 模拟输出通道驱动，接受 PC 端 AI 引擎和 JACE8000 本地 Wire Sheet 的双重控制。以下逐一介绍各硬件模块的型号参数与系统中的功能定位。

**温湿度变送器**。采用壁挂式安装，内置温度传感元件和电容式湿度传感元件，温度量程 -20 至 60°C，湿度量程 0 至 100%RH，两路独立 4-20mA 电流输出分别对应温度和湿度信号。设备接入 IO-22U 的 UI-1（温度）和 UI-2（湿度）通道。温度数据用于风扇控制逻辑中的启停判据和 FSM 态势评分中的温度变化率计算，湿度数据用于环境监测和 FSM 态势评分中的湿度变化率计算。

>

**光照传感器**。采用壁挂或吊顶安装，感光元件为硅光电二极管，光谱响应接近人眼视见函数，量程 0 至 2000 Lux，输出 4-20mA 电流信号。设备接入 IO-22U 的 UI-3 通道。光照度数据直接决定照明 LED 的补光策略——低于补光阈值时按模式亮度开启、高于阈值时关闭照明以实现自然光优先的节能控制，同时也作为 FSM 态势评分中光照变化率的计算依据。

**CO₂ 传感器**。采用非色散红外（NDIR）检测原理，量程 0 至 5000 ppm，精度 ±(50 ppm + 3% 读数)，输出 4-20mA 电流信号。设备接入 IO-22U 的 UI-4 通道。CO₂ 浓度是 FSM 态势评分中权重最高的信号（40%），也是风扇控制的双参数判据之一——浓度超过 1200 ppm 时无论温度如何均强制开启风扇以补充新风，浓度回落至 900 ppm 以下且温度达标时才允许关闭。

### 4.2 物联网传输层

本系统的数据传输链路跨越三个物理节点——IO-22U 扩展模块、JACE8000 边缘控制器和 PC 端 AI 服务器——其间经过两段协议转换与一次 API 封装。第一段为 IO-22U 与 JACE8000 之间的 RS-485 Modbus RTU 现场总线，负责将传感器模数转换后的数字量汇聚至边缘控制器；第二段为 JACE8000 与 PC 端之间的 oBIX over HTTPS 通信，JACE8000 作为 oBIX 服务端提供传感器数据的只读接口和输出设备数据的写入接口；第三层为 PC 端 Flask 服务器对前端 Dashboard 暴露的 RESTful JSON API，以自定义标准完成控制数据与人机交互界面的对接。以下按数据流向自底向上逐一阐述。

#### 4.2.1 JACE-内部数据管理与通信逻辑

JACE8000 控制器作为边缘核心，负责管理其直接连接的 IO-22U 模块及其上的全部传感器和执行器。传感器采集到的原始数据首先通过 IO-22U 的 8 路通用输入通道完成模数转换，经 RS-485 Modbus RTU 总线（波特率 38400 bps、8 位数据位、无校验、1 位停止位）以 IO-22U 为地址 1 从站、JACE8000 为主站的方式周期性上传至 JACE8000 内部。在 JACE8000 的 Niagara Framework 4 环境下，这些物理输入被映射为可读可写的"点"对象，形成统一的数据模型——每个传感器对应一个 Niagara Point，包含数值（out）、状态（status）、单位（unit）和告警（alarm）等标准化属性，所有点位在 Niagara 的 Points 容器中以层级命名空间组织，便于 oBIX 协议通过 URI 路径直接寻址。

JACE8000 能够作为 oBIX 服务端运行，在 Niagara 中启用 ObixService 组件后，自动将其内部管理的全部点位以 oBIX 对象树的形式对外暴露。传感器点位对外提供只读接口——外部客户端可通过 HTTP GET 请求读取点位实时值和状态元数据；输出设备点位对外提供写入接口——外部客户端可通过 HTTP POST 请求修改点位的目标值，由 Niagara 内部的输出驱动链将数字指令转换为对应通道的物理信号。这种以点位为中心的抽象方式为数据在 JACE8000 与 PC 端 AI 引擎之间的流通奠定了基础：上层应用无需关心底层是 Modbus RTU 寄存器还是 4-20mA 模拟量，仅需通过统一的 oBIX URI 即可完成读写操作。

#### 4.2.2 JACE-云端智控

对于 JACE8000 到 PC 端后台控制系统的信息交互，本方案选择 oBIX 应用协议作为通信标准。oBIX（Open Building Information eXchange）是由 OASIS 组织制定的、基于 RESTful Web 服务和 XML 的建筑控制系统开放接口标准，通过统一的对象模型、URI 命名、合约机制及可扩展的 Facets，使多种楼宇子系统（HVAC、照明、安全等）与企业级应用之间实现互操作和数据集成。选择 oBIX 而非直接使用 Modbus TCP 是本方案的重要技术决策——oBIX 的 URI 路径携带了点位名称、层级和单位的完整语义信息，Python 端仅需标准 HTTP 库即可读写任意点位值，无需处理寄存器地址映射、字节序转换和 Modbus 异常码解析；同时 oBIX 的 RESTful 架构天然兼容 HTTP 生态下的缓存、代理和负载均衡基础设施。

oBIX 基于客户端/服务器模型，遵循 REST 架构风格。URI 命名方面，所有 oBIX 对象由 URI 标识，根对象使用绝对 URI，其余使用相对 URI，通过 HTTP GET/PUT/POST/DELETE 实现资源访问。请求类型方面，Read 对应 HTTP GET（读取点位当前值）、Write 对应 HTTP PUT（写入点位目标值）、Invoke 对应 HTTP POST（调用点位支持的操作）、Delete 对应 HTTP DELETE。oBIX 以 obj 为根对象类型，定义一组原始对象类型并在 XML 中一一映射元素类型。本系统中各传感器和执行器的 oBIX 对象定义如表 4-3 所示。

**表 4-3  传感器与执行器 oBIX 对象定义**

| 传感器 | XML 类型 | 数值单位 | 描述 |
|--------|---------|---------|------|
| 温度传感器 | `<real>` | °C | 室内温度，范围 -20 至 60°C |
| 湿度传感器 | `<real>` | %RH | 室内湿度，范围 0 至 100%RH |
| CO₂ 传感器 | `<real>` | ppm | CO₂ 浓度，范围 0 至 5000 ppm |
| 噪声传感器 | `<real>` | dB | 环境噪声，范围 30 至 120 dB |
| 光照传感器 | `<real>` | Lux | 自然光照度，范围 0 至 2000 Lux |
| 烟雾传感器 | `<real>` | ppm | 烟雾浓度，范围 0 至 2000 ppm |
| PM2.5 传感器 | `<real>` | µg/m³ | 颗粒物浓度，范围 0 至 1000 µg/m³ |

| 风扇 | `<bool>` | — | 继电器通断，true=开启、false=关闭 |

在通信安全方面，oBIX 通信叠加 SSL/TLS 加密，以 HTTPS 协议传输。JACE8000 平台采用平台认证与站点认证双重身份验证机制，两个层级相互独立。平台身份认证对应 Niagara Daemon 层面的高权限访问——用户通过 Workbench 客户端首次连接 Platform 时，需输入平台级账号和密码，认证通过 SSL/TLS 加密在端口 5011 上完成，该凭据独立于站点用户管理。站点身份认证对应 Station 运行态的数据访问——成功连接 Platform 后，打开具体 Station 时需再次输入该 Station 的登录凭据，用户权限可细分为只读、写入或管理员级别，确保运行时的访问控制粒度。两层认证分别作用于设备管理面和数据操作面，配合 oBIX 协议和 TLS 加密通道，有效保障楼宇系统在物联网环境下的通信安全性与身份可追溯性。

| `/api/v1/assistant/history` | GET | 获取对话历史 |
| `/api/v1/assistant/quick-prompts` | GET | 获取快捷提问列表 |

后端采用线程分离架构实现控制逻辑与 API 服务的松耦合。控制主线程专注于核心任务循环——传感器数据采集、数据库写入、FSM 状态检测与设备控制——以固定 2 秒周期运行，确保环境响应的实时性和确定性；API 服务线程通过 Flask 开发服务器异步运行于独立线程中，与控制主线程形成松耦合关系。两个线程之间通过全局共享的 BackendRuntime 对象和 SQLite 数据库实现跨模块数据交换：传感器实时数据通过 `runtime.get_latest_sensor_payload()` 方法从控制线程的内存缓存中直接读取，避免了数据库查询的 I/O 延迟；历史数据和告警事件通过 DatabaseManager 从 SQLite 数据库中以只读连接查询，避免与控制线程的写入操作产生锁冲突；设备状态通过 `runtime.get_device_status()` 方法从 DeviceController 的内存状态字典中读取。这种共享内存 + 数据库双通道的数据访问模式，使得 API 服务在高并发查询压力下仍能保持毫秒级响应，而控制线程的 2 秒采集周期不受任何影响。

为保证系统在开放网络环境中的安全性，API 服务施加以下安全控制策略。CORS 跨域资源共享白名单控制，仅允许来自本机前端页面所在域（localhost:5000）的跨域请求，拒绝外部域的 AJAX 调用。请求频率限制，通过 Flask 扩展对每个客户端 IP 实施每秒不超过 60 次的请求速率限制，防止恶意高频调用和拒绝服务攻击。传输层加密，生产环境中 API 服务通过反向代理（如 Nginx）叠加 SSL/TLS 证书实现 HTTPS 加密传输，确保传感器数据、设备控制指令和用户对话内容在公网链路上的机密性。敏感信息保护方面，LLM API 密钥、oBIX 登录凭据和天气 API 密钥等敏感配置全部通过环境变量或 `.env` 文件注入，不写入版本控制系统或代码仓库。

### 4.3 物联网控制层

物联网控制层是系统将感知数据转化为智能决策的核心环节，部署于 PC 端的 Python 运行环境中，通过 oBIX over HTTPS 协议与 JACE8000 进行双向数据交换。与感知层侧重硬件接入不同，控制层的设计重点在于如何在保证控制确定性和安全性的前提下引入 AI 智能决策能力。为此，本系统采用"双引擎协作"的软件架构——以本地规则引擎为执行主体确保控制指令的实时性和可靠性，以 LLM 顾问为辅助参考提供基于多因素推理的环境优化建议。控制层以独立后台线程运行主控制循环（周期 2 秒），Flask Web 服务运行于主线程，两者通过共享内存对象和 SQLite 数据库实现松耦合数据交换。以下依次阐述整体软件架构、FSM 状态机、LLM 顾问、安全规则和舒适性控制五个核心模块的实现细节。

#### 4.3.2 FSM 状态机实现

FSM 状态机的实现由两个 Python 类协作完成：`OccupancyDetector` 负责从传感器快照中计算综合态势值 S_t，`OccupancyFSM` 负责根据 S_t 执行状态迁移。

`OccupancyDetector` 类的核心属性为 `previous_snapshot` 字典，在每轮评估中保存上一周期的传感器快照，用于计算相邻采样间的环境变化量。公开方法 `evaluate(snapshot)` 接收包含全部传感器当前读数的字典，按三步执行：第一步调用 `_score_snapshot(snapshot, previous)` 计算当前态势值；第二步调用 `self.fsm.update(score)` 更新状态机；第三步将当前快照复制为 `previous_snapshot` 供下一轮使用。返回值包含迁移前的旧状态、迁移后的新状态和当前 S_t 数值的三元组。

`_score_snapshot` 方法首先计算三个派生特征。若 `previous` 参数非空，取当前温度与上一轮温度的绝对差作为温度变化量，同理计算湿度变化量和光照变化量——这三个变化量分别乘以各 2 秒的采样间隔，单位为 °C/2s、%RH/2s 和 Lux/2s，反映的是人员活动引起环境的瞬时微扰而非传感器读数的绝对值。然后将五个特征分别送入 `_normalize(value, key)` 方法归一化：该方法从 `OCCUPANCY_SCORE_RANGES` 配置字典中查找对应键的 [low, high] 区间，值低于 low 返回 0.0、高于 high 返回 1.0、中间值按 `(value - low) / (high - low)` 线性映射。五个归一化结果按 FSM_WEIGHTS 中预设的权重进行加权求和——CO₂ 40%、噪声 25%、温度变化率 10%、湿度变化率 10%、光照变化率 15%——最终经 `max(0.0, min(1.0, score))` 钳位至 [0, 1] 区间，得到综合态势值 S_t。

`OccupancyFSM` 类维护一个字符串类型的实例变量 `state`，初始值为 `VACANT`。公开方法 `update(score)` 接收 S_t 数值，按当前状态分四种情况执行迁移判断。VACANT 态：S_t ≥ 0.45 直接迁移至 OCCUPIED（快速来人场景，跳过过渡态），S_t 介于 0.40 和 0.45 之间迁移至 ARRIVING。ARRIVING 态：S_t ≥ 0.45 迁移至 OCCUPIED（确认有人），S_t 回落至 FSM_ENTER_ARRIVING × 0.5 = 0.20 以下退回 VACANT（排除瞬时扰动误判）。OCCUPIED 态：S_t < 0.40 迁移至 LEAVING（人员开始离开的信号）。LEAVING 态：S_t ≤ 0.30 迁移至 VACANT（确认人员已稳定离开），S_t 重新回升至 0.45 以上则返回 OCCUPIED（人员中途返回场景）。所有迁移判断从上到下按所列顺序执行，一旦命中即返回——这意味着 VACANT 态下如果 S_t 同时满足 ≥ 0.45 和 ≥ 0.40 两个条件，优先执行直接跳至 OCCUPIED 的分支。`update` 方法的返回值为迁移前的旧状态字符串，供上层 BackendRuntime 记录状态变更事件。

从 VACANT→ARRIVING（0.40 进入）到 OCCUPIED（0.45 进入）的 0.05 间隙、从 OCCUPIED→LEAVING（0.40 退出）到 LEAVING→VACANT（0.30 最终退出）的 0.10 间隙，共同构成了系统在"有人"和"无人"之间的非对称滞回带。这一设计的物理基础在于 CO₂ 浓度的扩散惰性——人员进入后 CO₂ 逐步上升穿越 S_t 阈值需要一定时间，人员离开后 CO₂ 缓慢消散同样需要一定时间——FSM 的滞回阈值恰好与这一物理时间常数对齐，使得状态迁移与真实的物理过程同步，从根本上抑制了传感器噪声和瞬时扰动引发的误翻转。

#### 4.3.3 LLM 顾问实现

`LLMService` 类实现两大功能：面向控制循环的 `decide` 方法提供基于 LLM 的控制建议生成，面向 AI 对话助手的 `chat` 方法提供基于上下文的自然语言响应。

`decide` 方法的调用链遵循"守卫判断 → LLM 调用 → 解析校验 → 照度保护"四步流程。守卫判断阶段，方法首先检查 FSM 状态：若为 VACANT，直接返回 `ControlDecision(lighting_brightness=0, fan_state=False, reasoning="空间当前判定为无人状态……", source="fsm_fallback")`，不消耗任何 API 调用。若为 ARRIVING 或 OCCUPIED，且 `LLM_ENABLED` 为 true 且 `LLM_API_KEY` 已配置，则进入 `_llm_decide` 方法尝试调用 LLM；否则直接进入 `_fallback_decide` 本地降级路径。

`_llm_decide` 方法构造两条消息发送至 DeepSeek API 的 `/chat/completions` 端点。system prompt 明确限定 LLM 的角色——"你是图书馆物联网舒适性控制助手。只为照明和风扇给出舒适性建议，不要处理烟雾、噪声等安全动作；安全动作由后端规则独立执行。必须只输出一个 JSON 对象，不要 Markdown。"——并在 schema 中约束输出 JSON 必须包含 `lighting_brightness`（0-100 整数）、`fan_state`（布尔值）和 `reasoning`（中文简短理由）三个字段。user message 以 JSON 格式传入全部上下文：传感器全量快照、当前 FSM 状态、偏好模式名称及对应阈值配置、室外天气数据（天气状况与温度）、以及规则约束（CO₂ 舒适上限 1200 ppm、VACANT 已本地处理等）。API 请求参数为 `temperature=0.1`（低随机性以保证输出一致性）、`max_tokens=300`（紧凑输出以控制延迟）和 `response_format={"type": "json_object"}`（结构化约束）。

LLM 返回的 HTTP 响应经 `_parse_json_object` 方法解析：首先尝试 `json.loads` 直接解析；若失败则以正则 `\{.*\}` 兜底提取 JSON 片段再解析；若仍失败或缺少必填字段则抛出异常，由上层 `decide` 方法的 `except` 块捕获后转入 `_fallback_decide`。校验通过的 `lighting_brightness` 值送入 `_guard_lighting_brightness` 方法进行本地照度保护——若当前实际光照度已高于当前偏好模式的补光阈值，LLM 建议的亮度被强制置零，并在 reasoning 中追加"后端照度保护：当前 XX Lux 已高于 XX Lux 补光阈值，强制关闭照明"的说明文本。这一机制确保 LLM 即便不了解当前实际照度情况而给出不合理的高亮度建议，也不会造成照明能耗浪费。风扇建议不通过本地保护——温度与 CO₂ 的最终双参数判别由 DeviceController 的规则引擎在指令执行阶段完成。

`_fallback_decide` 方法在 LLM 不可用时提供本地规则降级。照明的降级逻辑：直接取当前偏好模式的 `lighting_brightness` 预设值，经 `_guard_lighting_brightness` 照度保护后返回。风扇的降级逻辑：以 `温度 ≥ fan_on_above_c 或 CO₂ ≥ 1200` 为开启条件。降级推理文本包含当前 FSM 状态、偏好模式名称、各传感器当前数值和对应的控制建议，若因 LLM 异常触发降级则在文本前追加异常原因描述（如"LLM 不可用，已降级为本地规则：timeout"）。降级过程对控制循环完全透明——上层 BackendRuntime 仅看到统一的 `ControlDecision` 对象，不感知底层是 LLM 还是本地规则生成了决策。

`ControlDecision` 是一个 Python dataclass，包含四个字段：`lighting_brightness: int`（0-100）、`fan_state: bool`、`reasoning: str`（中文推理说明，将显示在 Dashboard 的 LLM 决策面板中）和 `source: str`（取值 `"llm"` 或 `"fsm_fallback"`，用于 dashboard 的 AI 模式指示灯颜色切换——绿色=LLM 活跃，黄色=FSM 降级）。

`chat` 方法为 AI 对话助手提供自然语言响应能力。方法接收用户文本消息、当前系统上下文字典（传感器数据、系统状态、设备状态、活跃告警、能耗摘要）和历史对话记录。内部通过关键词匹配自动路由至四个回答模板：包含"趋势"或"trend"时调用 `_trend_answer`，遍历各传感器的最近历史数据计算变化趋势和方向；包含"预案""应急"或"告警"时调用 `_emergency_answer`，根据活跃告警列表生成分级处理建议（烟雾告警→确认来源疏散人员、噪声告警→检查噪声源提醒降噪）；包含"能耗""节能"或"energy"时调用 `_energy_answer`，输出当前统计范围的耗电量、运行时长、平均功率和相对常开模式的节能百分比；其余通用环境咨询调用 `_environment_answer`，输出温度、湿度、照度、CO₂、噪声、烟雾和 PM2.5 的全量数据摘要以及基于阈值规则的具体建议（CO₂ 超标→加强通风、噪声超标→确认噪声源、照度不足→保持照明开启等）。所有回答模板均基于本地规则和数据生成，不产生 LLM API 调用开销。响应文本经 `stream_tokens` 方法按每 12 个字符切分为 Token 序列，通过 Flask SSE（Server-Sent Events）端点 `/api/v1/assistant/chat?stream=true` 逐块推送至前端，实现近似实时对话的打字机效果。

#### 4.3.4 安全规则实现

`DeviceController` 类在每轮控制循环的 `update` 方法中，首先调用 `_apply_safety_rules` 方法独立处理安全告警，再调用 `_apply_comfort_control` 方法处理舒适性控制——安全规则始终在舒适性控制之前执行，确保告警响应不受任何决策链路延迟的影响。`_apply_safety_rules` 方法接收三个参数：当前时间的 datetime 对象、ISO 格式时间戳字符串和传感器快照字典。方法内部按烟雾告警和噪声告警两套独立的判断-触发-解除逻辑并行执行。

烟雾告警的判断逻辑基于单一阈值比较。从快照中读取 `smoke` 值，若大于等于 `SMOKE_THRESHOLD`（150 ppm），立即将 `_smoke_below_since` 计时器清空，调用 `database.open_alert()` 向 alerts 表写入一条 `severity="critical"` 的告警记录，消息包含当前烟雾浓度实测值和触发阈值，同时通过 `_set_bool_device` 方法将蜂鸣器的目标状态设为 true——`_set_bool_device` 内部先比较目标状态与缓存的当前状态，仅在状态确实发生变化时才执行 oBIX 写入。若烟雾值回落至 `SMOKE_THRESHOLD - SMOKE_HYSTERESIS`（150 − 20 = 130 ppm）以下，系统启动 5 秒持续确认计时器：首次低于 130 ppm 时记录 `_smoke_below_since` 时间戳，后续每轮循环检查是否已持续满足 5 秒，满足后调用 `database.resolve_alert()` 将告警记录的 `resolved_at` 字段更新为当前时间，并通过 `_set_bool_device` 将蜂鸣器目标状态设为 false 解除告警。若在 5 秒计时期间烟雾值再次回升至 150 ppm 以上，计时器被清空并重新触发告警，计时从零开始。

噪声告警的判断逻辑引入了上升沿和下降沿两套非对称持续计时机制。从快照中读取 `noise` 值，若大于等于 `NOISE_THRESHOLD`（65 dB），将 `_noise_below_since` 清空，首次触发时记录 `_noise_above_since` 时间戳，后续每轮检查是否已持续超过 `NOISE_DURATION_SECONDS`（10 秒）——10 秒内噪声短暂回落至 65 dB 以下则 `_noise_above_since` 被清空，计时重置。只有连续 10 秒维持在阈值以上才判定为确认的噪声超标事件，调用 `database.open_alert()` 写入 `severity="warning"` 的告警记录，并通过 `_set_bool_device` 将警示 LED 的目标状态设为 true。下降沿方向，噪声值回落至 `NOISE_THRESHOLD - NOISE_HYSTERESIS`（65 − 7 = 58 dB）以下时，首次记录 `_noise_below_since` 时间戳，持续满足 `NOISE_CLEAR_DURATION_SECONDS`（20 秒）后方可解除告警并熄灭警示 LED——20 秒的解除确认时间远长于 10 秒的触发确认时间，防止噪声源间歇性发作时警示 LED 出现闪烁。两套计时器相互独立：上升沿计时期间若噪声回落则计时清零，下降沿计时期间若噪声回升则计时清零并可能重新进入上升沿判定。

安全规则在 JACE8000 本地 Wire Sheet 中同步配置了等效的控制逻辑——当 JACE8000 检测到 UI-6 烟雾传感器值 ≥ 150 ppm 时，直接在边缘侧闭合 DO-1 继电器驱动蜂鸣器，不经过 PC 端的任何处理环节。PC 端的 `_apply_safety_rules` 与 JACE8000 的 Wire Sheet 安全规则形成双通道冗余——两者并发执行、互不依赖、任一失效不影响另一通道的告警输出。

#### 4.3.5 舒适性控制实现

舒适性控制由 `_apply_comfort_control` 方法实现，在每轮控制循环中于安全规则之后执行，接收时间戳、传感器快照和 FSM 状态三个参数。方法的核心逻辑分为四个阶段：获取偏好模式配置、计算本地规则控制值、调用 LLM 顾问获取建议、执行设备写入。

第一阶段，从 `config.PROFILES` 字典中读取当前 `active_profile` 对应的参数配置——`fan_on_above_c`（风扇开启温度阈值）、`light_on_below_lux`（照明补光阈值）和 `lighting_brightness`（照明亮度百分比）。若 FSM 状态为 VACANT，将照明亮度和风扇状态直接置为零和关闭，跳过后续所有阶段——VACANT 态的强制关闭策略由此在代码层面硬编码保障。若状态为非 VACANT，则进入第二阶段。

第二阶段调用两个规则方法计算本地控制值。`_rule_lighting_brightness(snapshot, profile)` 方法执行一条简洁的条件判断：当前光照度 ≥ 模式补光阈值时返回 0（自然光充足，不开灯），否则返回模式预设的 `lighting_brightness` 值（节能 50、均衡 60、舒适 80），结果经 `max(0, min(100, ...))` 钳位。`_guard_fan_state(current_state, snapshot, profile)` 方法实现温度-CO₂ 双参数回滞控制——温度 ≥ 模式风扇开启温度或 CO₂ ≥ 1200 ppm 时返回 true（开启）；温度 ≤ 开启温度 − `FAN_TEMP_HYSTERESIS`（1°C）且 CO₂ ≤ `CO2_FAN_OFF_BELOW`（900 ppm）时返回 false（关闭）；两者之间的回滞区间内返回 `current_state`（维持当前状态不变）。1°C 温度回滞和 300 ppm CO₂ 回滞共同防止风扇在阈值边界反复启停。

第三阶段调用 `self.llm_service.decide()` 获取 LLM 顾问的 ControlDecision 对象。LLM 的 lighting_brightness 和 fan_state 建议不直接用于设备控制——仅将其 `reasoning` 文本和 `source` 来源标识保存至 `self.last_llm_decision` 和 `self._last_comfort_source` 字段，供 Dashboard 的 AI 模式指示灯和 LLM 决策面板展示。实际控制值完全由第二阶段的本地规则计算结果决定，LLM 的角色被严格限定为"顾问"。

第四阶段执行设备写入。若照明 LED 处于 auto 模式，调用 `_set_lighting(timestamp, brightness, source)` 方法——该方法内部将 0-100 的亮度百分比线性映射为 0-10V 的模拟电压值（`brightness / 100 × LIGHTING_ANALOG_MAX`，其中 `LIGHTING_ANALOG_MAX` = 10.0V），根据 `DEVICE_POINTS` 中照明 LED 的 `kind` 字段（bool 或 real）决定写入物理值的类型，然后调用 `_write_obix_value` 通过 oBIX 协议下发至 JACE8000。若风扇处于 auto 模式，调用 `_set_bool_device(timestamp, device, desired_state, source, trigger)` 方法下发风扇启停指令。`_set_lighting` 和 `_set_bool_device` 方法均内置状态缓存比较——若目标亮度/状态与缓存的当前值一致，则跳过 oBIX 写入以避免冗余通信。

`_write_obix_value` 是所有设备写入操作的统一出口。该方法首先检查当前运行模式：若处于仿真模式（`SIMULATION_MODE=True`）或目标设备标记为未安装，则跳过实际的 oBIX 调用，仅记录设备事件日志；否则调用 `obix_client.write_point(point_name, value, kind)` 将指令以对应 XML 格式（bool 类型输出 `<bool name="out" val="true/false"/>`，real 类型输出 `<real name="out" val="..."/>`）通过 HTTPS POST 发送至 JACE8000 的 oBIX 端点。oBIX 写入若发生异常，异常信息被记录至 `self.last_write_error` 并将 `self.degraded` 标志置为 true，触发 Dashboard 的降级状态提示。无论写入成功与否，该方法在 finally 块中调用 `database.log_device_event()` 将本次设备操作的时间、设备名、动作、参数值和触发源持久化写入 device_events 表。

每次控制指令发生变化时——即照明亮度、风扇状态和 FSM 状态三元组与上一轮不同——系统调用 `database.log_llm_decision()` 将决策记录写入 llm_decisions 表，包含时间戳、FSM 状态、完整推理文本和控制动作 JSON，用于后续能耗统计和决策回溯。决策签名缓存 `self._last_decision_signature` 确保仅在指令确实变化时才写入数据库，避免冗余记录。

### 4.4 应用层与可视化

应用层位于物联网四层架构的最顶端，直接面向最终用户，负责将感知层采集的环境数据、控制层生成的决策结果以直观、实时的方式呈现给管理者，同时提供设备手动控制和历史数据回溯的能力。本系统的应用层由三部分构成：SQLite 轻量级数据库负责所有运行数据的持久化存储，Flask Web 服务器提供 RESTful API 接口和静态页面服务，基于 HTML5 Canvas 和原生 JavaScript 构建的浏览器端 Dashboard 则提供包括主控看板、AI 助手、3D 可视化、历史趋势和能耗分析在内的五个核心页面。三者构成了"数据存储—接口服务—用户界面"的完整应用栈，所有前后端通信均通过 JSON 格式的 RESTful API 完成。以下分别介绍数据库设计、Web 服务架构和前端页面的具体实现。

#### 4.4.1 数据库设计

系统采用 SQLite 轻量级数据库存储运行时数据，启用 WAL 模式以支持并发读写。数据库文件名为 `cloud_controller.db`，位于后端程序同级目录。包含五张核心数据表：

- **sensor_readings**：传感器时序数据，包含时间戳、传感器名、数值和单位字段，按传感器名和时间戳建立复合索引
- **device_events**：设备操作事件日志，包含时间戳、设备名、操作、参数和来源标识字段
- **fsm_state_log**：FSM 状态迁移日志，包含时间戳、状态名和态势评分字段
- **llm_decisions**：LLM 决策记录，包含时间戳、FSM 状态、推理文字和控制动作 JSON 字段
- **alerts**：告警记录，包含触发时间、解除时间、类型、严重级别、消息、传感器值和阈值字段

>
> 标准 Crow's Foot ER 图，5 张表各用一个矩形框表示——框头用深色填充标注表名，框体列出字段名，左侧用钥匙图标标注主键（PK），菱形图标标注外键依赖。表之间的关联线采用不同的线型和端点符号：实线+鱼尾纹=一对多关系（sensor_readings 到 fsm_state_log 等），虚线=逻辑引用关系（sensor_readings 到 alerts 通过 sensor_name 和 type 的逻辑关联）。五张表按使用频率从左到右排列：sensor_readings（最大最宽，核心表）居中，device_events 和 alerts 分别位于左下方和右下方（操作型表），fsm_state_log 和 llm_decisions 位于上方两侧（日志型表）。每张表背后用不同大小的半透明圆形表示数据量级（sensor_readings 背后圆最大）。底部用一条浅色横条标注数据库配置信息（SQLite WAL 模式、ISO 8601 时间戳格式、复合索引名称），用齿轮+数据库小图标代替纯文字。

#### 4.4.2 Web 服务架构

`web_server.py` 是 Flask Web 服务器的入口模块，采用应用工厂模式构建——模块顶层的 `create_app(runtime)` 函数接收 `BackendRuntime` 实例作为依赖注入参数，内部通过 `@app.route` 装饰器注册全部路由处理函数，最终返回配置完毕的 Flask 应用实例。应用级配置包括 `app.json.ensure_ascii = False` 以确保 JSON 响应中原生输出中文字符（避免 Unicode 转义），以及两个静态文件目录的路径解析——`frontend_dir` 指向 HTML/CSS/JS 前端资源目录，`icon_dir` 指向图标资源目录。

API 路由统一挂载在 `/api/v1` 基础路径下，按功能域分为六个端点组。传感器数据组包含三个端点：`GET /api/v1/sensors/latest` 调用 `runtime.get_latest_sensor_payload()` 返回全部传感器的最新读数、采集元信息和传感器在线诊断状态，响应 JSON 结构分为顶层时间戳、data 子对象（每个传感器一个键，包含 value 和 unit 字段）和 meta 子对象（采集来源、循环错误、成功轮询次数和逐传感器在线状态）；`GET /api/v1/sensors/history` 接收 `sensor` 查询参数指定传感器名、`start` 和 `end` 查询参数指定 ISO 8601 时间范围（未提供则默认最近 1 小时），调用 `runtime.get_sensor_history()` 从 SQLite 的 sensor_readings 表中按传感器名和时间范围查询时序数据，返回包含传感器名、单位和 data 数组（每个元素含 ts 和 value）的 JSON 对象；`GET /api/v1/sensors/diagnostics` 返回采集循环的诊断信息，包括最后一次循环错误、成功轮询次数和每个传感器的在线状态布尔值。

设备控制组包含四个端点：`GET /api/v1/devices/status` 调用 `runtime.get_device_status()` 返回四个执行器的当前状态字典——buzzer 和 warning_led 包含 state（布尔值）和 trigger（触发源名称或 null）字段，lighting_led 额外包含 brightness（0-100 整数）字段，fan 包含 mode（auto/manual）字段；`GET /api/v1/devices/events` 接收可选的 `limit`（默认 50）和 `device` 查询参数，调用 `runtime.get_device_events()` 从 device_events 表中按设备名筛选和按条数限制返回操作事件日志列表，每条事件包含时间戳、设备名、动作、参数值和触发源；`POST /api/v1/devices/control` 接收 JSON 请求体中的 `device`、`action` 和可选的 `value` 字段，调用 `runtime.control_device()` 执行手动覆盖操作——buzzer 和 warning_led 支持 on/off/auto，lighting_led 支持 set_brightness（需附带 value 为 0-100 整数）/auto，fan 支持 on/off/auto——方法内部校验设备名和动作的合法性，非法请求返回 400 错误；`GET /api/v1/devices/diagnostics` 返回设备写入的诊断信息，包括最近一次 oBIX 写入错误和设备降级状态标志。

系统管理组包含三个端点：`GET /api/v1/system/profiles` 调用 `runtime.get_profiles()` 返回当前激活的偏好模式名称和全部三种模式的完整配置参数（风扇开启温度、照明补光阈值、照明亮度百分比）；`POST /api/v1/system/profile` 接收 JSON 请求体中的 `profile` 字段（可选值 energy_saving/balanced/comfort），调用 `runtime.set_profile()` 切换模式，非法模式名返回 400 错误，切换成功后返回更新后的模式信息；`GET /api/v1/system/status` 调用 `runtime.get_system_status()` 返回系统运行全貌——FSM 当前状态和态势评分值、FSM 手动覆盖状态（Demo 模式）、AI 决策模式（llm_advice/fsm_fallback/emergency_override/manual_override）、最近一次 LLM 推理文本、设备降级状态标志和传感器数据采集来源（obix/simulation）。Demo 快捷操作组的 `/api/v1/shortcuts` 和 `/api/v1/shortcuts/action` 端点提供竞赛演示用的一键环境切换功能——action 支持 set_fsm（直接覆写 FSM 状态和评分，如设为 OCCUPIED+0.72）和 clear_fsm（清除覆写恢复自动检测），以及模拟告警的触发与解除，方便评委现场验证系统对不同场景的响应。

能耗统计组包含两个端点：`GET /api/v1/energy/summary` 接收 `range` 查询参数（hour/day/week），调用 `runtime.get_energy_summary()` 返回指定时间范围内的累计耗电（kWh）、风扇运行时长（分钟）、平均功率（W）、CO₂ 减排量（kg）以及 AI 模式与假设常开模式的能耗对比（含节能百分比）；`GET /api/v1/energy/timeseries` 按相同范围参数返回能耗时序数据数组，每个元素包含时间戳 ts、风扇功率 fan_power_w 和风扇状态 fan_state 三个字段，供前端能耗趋势图表绑制。告警组的 `GET /api/v1/alerts` 调用 `runtime.get_active_alerts()` 返回 alerts 表中所有未解除告警（resolved_at 为 NULL）的列表，每条包含告警类型、严重级别（critical/warning）、触发消息、传感器实测值和触发阈值。

AI 助手组包含三个端点：`GET /api/v1/assistant/quick-prompts` 返回四个预置的快捷提问文本（分析当前环境数据、查看近期趋势、查询应急预案、分析当前能耗），每个包含 id 和 text 字段；`GET /api/v1/assistant/history` 接收 `limit` 查询参数（默认 20），调用 `runtime.get_assistant_history()` 返回最近对话记录列表；`POST /api/v1/assistant/chat` 接收 JSON 请求体中的 `message` 和可选的 `stream` 字段——若 `stream` 为 false（默认），调用 `runtime.chat_with_assistant(message)` 同步返回完整的 JSON 响应；若 `stream` 为 true，则返回 `text/event-stream` 类型的 SSE（Server-Sent Events）流式响应，通过 Flask 的 `stream_with_context` 生成器配合 `Response` 对象实现。生成器函数内部遍历 `runtime.stream_assistant_response(message)` 返回的 Token 序列，每个 Token 封装为 `data: {"token": "..."}\n\n` 格式的 SSE 消息帧，序列末尾发送 `data: [DONE]\n\n` 结束信号。响应头部设置 `Cache-Control: no-cache` 禁止代理缓存以确保流式文本的实时性。

前端静态文件通过三个路由对外服务：`/app` 执行 302 重定向至 `/app/`（确保路径末尾斜杠一致性）；`/app/` 返回 `index.html` 主页面文件；`/app/icon/<filename>` 从 icon_dir 目录返回图标资源；`/app/<path:filename>` 以 `send_from_directory` 从 frontend_dir 目录返回任意路径的静态文件（CSS、JS、图片等）。`send_from_directory` 由 Flask 内置提供，自动处理 MIME 类型推断、缓存头设置和路径安全校验。

路由处理函数中的通用模式为：查询参数通过 `request.args.get()` 获取并以默认值兜底，非法参数返回 `{"error": "..."}` 的 JSON 400 响应；请求体通过 `request.get_json(silent=True)` 静默解析（不自动返回 415 错误），解析失败或必填字段缺失时返回描述性错误信息；所有来自 `runtime` 的业务异常（如 `ValueError`）被 try/except 捕获后转换为结构化的 JSON 错误响应而非抛出 500 内部错误。辅助函数 `_parse_datetime(raw)` 位于模块底部，使用 `datetime.fromisoformat()` 解析 ISO 8601 时间字符串，若解析结果不带时区信息则自动附加系统本地时区，确保时间范围查询在不同客户端的时区差异下结果一致。

### 5.3 总结与展望

本项目基于霍尼韦尔 Niagara Framework 4 + JACE8000 + IO-22U 技术栈，成功设计并实现了一套面向高校图书馆的 AI 自主优化低碳管控系统。系统以"本地规则引擎为主、LLM 顾问为辅"的双引擎架构，在保证控制安全性和确定性的前提下引入了 AI 智能决策能力，并以三层降级保底机制确保系统在任何故障模式下均能维持基础运行。系统全链路使用工业级传感器和 Modbus RTU 与 oBIX 标准开放协议，具备模块化软件架构，兼顾智能化水平与工业级可靠性。

未来可从以下方向持续优化和扩展：

- **LLM 增强**：当 LLM API 稳定性进一步提升后，可在安全边界内增加 LLM 建议的采纳权重，使系统从"规则执行 + AI 参考"向"AI 优化 + 规则兜底"平滑演进
- **多区域扩展**：将单区域控制扩展至图书馆多个功能区，如阅览区、自习区和书库区，各区域独立 FSM 判断、协同优化
- **历史数据分析**：基于长期传感器历史数据和设备运行日志，利用 LLM 进行周期性趋势分析并输出节能优化报告
- **课表联动**：对接图书馆开放时间或学校课表系统，实现从被动"感知-响应"到主动"预测-预备"的控制升级
- **用户偏好自学习**：基于手动控制操作历史记录，学习特定时段和传感器条件下的用户偏好，逐步减少人工干预
